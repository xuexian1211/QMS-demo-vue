## Context
QMS 系统需要与 WMS、MES 和 ERP (如金蝶云星空) 等外部系统深度集成，以便在关键业务节点（如采购收货、生产下达、产品入库、发货审核等）自动触发适当维度的 IQC、IPQC、FQC 或 OQC 检验任务。目前我们已完成了检验计划和方案这两个基座，接下来是“引擎”调度。不同来源系统的请求参数和触发时机各异。

## Integration Scenarios & Internal Logic

### 1. 各业务系统对接场景
* **WMS (IQC)**：在“采购收货单”提交后调用，触发 IQC 任务。
* **MES (IPQC)**：在“生产指令单”下达或开工报工时调用，触发 IPQC 任务。
* **WMS (FQC)**：在“生产入库单”生成或产品下线扫码时调用，触发 FQC 任务。
* **金蝶 ERP (OQC)**：在“销售发货单”审核或提交时调用，触发 OQC 任务。

### 2. QMS 内部匹配与生成逻辑
当 QMS 接收到上述接口请求后，将按以下流程自动匹配并生成任务：

* **第一步：计划匹配 (Plan Matching)**
  * 系统根据请求中的 `org_id`、`inspect_type` 和 `material_code` 在检验计划模型中检索匹配的生效计划。
  * **多维度过滤**：结合 `ext_context` 中的供应商或客户信息，匹配计划中的“适用范围”。
  * **优先级排序**：若匹配到多个计划，按计划预设的“优先级”（如 P5 > P20）执行。
* **第二步：触发条件判定 (Condition Check)**
  * 检查计划中的“触发配置”。
  * **事件驱动**：校验单据类型与状态是否一致（如“采购收货单-已提交”）。
  * **特殊策略**：检查是否满足“新供应商前 N 批”等动态策略。
* **第三步：任务实例化 (Task Generation)**
  * **引用方案快照**：锁定该时刻“关联方案”中的检验项和抽样规则。
  * **参数覆盖应用**：若计划中配置了“参数覆盖”（如：由于 A 线精度要求更高，将平面度公差从 0.05 修改为 0.03），则以计划中的值生成任务标准。
  * **指派执行人**：自动将任务分派给计划中设定的“默认检验员”。

## Goals / Non-Goals
- **Goals**:
  - 提供 `POST /api/v1/inspection/task/trigger`。
  - 基于计划匹配、条件校验对匹配到的计划进行实体化（产生任务实例）。
  - 提供可视化的模拟测试界面和详细的触发日志。
- **Non-Goals**:
  - 处理检验任务执行过程的详细业务（不在本次自动化生成范畴内）。
  - 处理与外部系统建立实际异步消息服务 (如 MQ) 的下层通信实现，目前仅支持 HTTP API。

## Decisions
- **Decision**: 使用单一统一入口 `/api/v1/inspection/task/trigger`。
- **Alternatives considered**: 针对每个业务场景提供单独入口 `/api/iqc/trigger`, `/api/ipqc/trigger` 等。
- **Rationale**: 单一入口利于日志聚合追踪及底层匹配核心逻辑复用。具体类型可以通过 `inspect_type` 和 `source_bill_type` 等字段区分。

## Risks / Trade-offs
- **Risk**: 复杂的动态策略覆盖判断可能会对接口性能造成影响。
  - **Mitigation**: 接口内应进行预校验，或考虑将耗时的匹配生成任务转换为异步处理机制，先记录任务请求再后台排队生成（当前视数据量先采取同步直出，记录性能指标）。
  
## Migration Plan
- 数据库新增触发测试日志表（`ins_trigger_log`）和控制任务生成的特定相关配置表维护。
