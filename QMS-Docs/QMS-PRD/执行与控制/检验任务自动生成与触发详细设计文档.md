# 检验任务自动生成与触发机制详细设计文档

## 1. 概述
在 QMS （质量管理系统）与外部系统（WMS仓储管理系统、MES制造执行系统、ERP金蝶云星空系统等）交互的闭环生态中，检验任务的自动生成机制是关键的枢纽。本设计文档描述了QMS如何提供统一的业务事件接收接口，自动匹配系统中已发布的检验计划模型，并自动实例化检验任务的过程。同时也包括为了调试和追溯而设计的“触发模拟器”和“任务触发日志”两部分页面模块。

## 2. 数据模型设计

### 2.1 检验任务触发日志表 (`qms_trigger_log`)
用于记录外部系统每一次调用触发接口的原始报文、判定链路和最终结果。

| 字段名 | 类型 | 说明 | 约束 |
|---|---|---|---|
| id | bigint | 主键 ID | PK |
| request_id | varchar(64) | 请求唯一标识 (幂等性控制) | Unique |
| request_time | datetime | 请求触发时间 | Not Null |
| source_system | varchar(32) | 来源系统 (WMS/MES/ERP) | Not Null |
| source_bill_type | varchar(64) | 来源单据类型 (如 PUR_RECEIVE) | Not Null |
| source_bill_no | varchar(64) | 来源单据号 | Not Null |
| inspect_type | varchar(32) | 检验类型 (IQC/IPQC/FQC/OQC) | Not Null |
| material_code | varchar(64) | 物料编码 | Not Null |
| batch_no | varchar(64) | 批次号 | Not Null |
| org_id | varchar(32) | 组织/工厂 ID | Not Null |
| ext_context | json | 扩展上下文 (客户, 供应商, 产线等) | Nullable |
| judgment_status | varchar(20) | 判定状态 (SUCCESS, IGNORED, FAILED) | Not Null |
| matched_plan_code | varchar(64) | 匹配到的计划编号 | Nullable |
| generated_task_no | varchar(64) | 生成的任务编号 | Nullable |
| detail_reason | varchar(500) | 判定详细原因 | Nullable |
| raw_payload | json | 原始请求报文 | Nullable |
| create_time | datetime | 创建时间 | Not Null |

### 2.2 核心相关表关联说明
- **检验计划表 (`qms_insp_plan`)**: 作为匹配源。包含 `org_id`, `insp_type`, `material_code`, `status`, `trigger_bill_type`, `priority`, `ext_rules` 等字段。当接口收到请求，会在本表进行多维度严格匹配和打分排序。
- **检验任务表 (`qms_insp_task`)**: 作为生成结果。当触发匹配成功且条件符合时，会在本表中插入数据（即`generated_task_no`对应的记录），包含基础信息并关联检验项内容。
- **检验方案表 (`qms_insp_scheme`)**: 在计划匹配成功后，系统锁定该计划关联的最新版本或指定版本的检验方案，将方案中的检验条目（`qms_insp_item`）转化为具体的任务实测项。

## 3. 统一触发接口设计

为方便外部系统接入与后期维护（所有检验场景复用），QMS 仅开放一个统一的业务事件接收接口。

* **接口地址**: `POST /api/v1/inspection/task/trigger`
* **接口协议**: HTTP/JSON
* **鉴权方式**: API Key / 内部服务调用鉴权 Token

### 3.1 请求报文 (Payload)

| 字段名 | 类型 | 必填 | 含义说明 | 示例 |
|---|---|---|---|---|
| orgId | String | 是 | 组织/工厂 ID (对应计划归属组织) | HFC_001 |
| inspectType | String | 是 | 检验类型 (IQC/IPQC/FQC/OQC) | IQC |
| sourceBillType | String | 是 | 来源单据类型 (控制触发源) | PUR_RECEIVE |
| sourceBillNo | String | 是 | 来源单据号 (外部单据唯一号) | WMS-REC-20260226 |
| materialCode | String | 是 | 物料编码 | M002 |
| batchNo | String | 是 | 批次号/序列号 | B20260226-01 |
| extContext | Map<String,String> | 否 | 扩展维度信息 (如供应商、产线、客户) | `{"supplier":"S01","line":"A01"}` |

### 3.2 响应报文 (Response)

| 字段名 | 类型 | 含义说明 | 示例 |
|---|---|---|---|
| code | Integer | 状态码 (200=请求成功接收处理) | 200 |
| message | String | 结果描述文字 | 操作成功 |
| data | Object | 详细结果对象 | 见下方说明 |

**data 对象说明**:
- `status`: 触发结果 `SUCCESS` / `IGNORED` / `FAILED`
- `taskNo`: 成功时生成的任务号 (如 `TSK-IQC-2026...`)
- `detail`: 详细的处理缘由说明。

## 4. 业务系统对接场景详细说明

### 4.1 WMS → IQC (来料检验)
- **触发节点**: 供应商送货，仓储人员在 WMS 中完成“采购收货单” 或 “到货点收单” 的确认提交操作。
- **参数传递**:
  - `inspectType`: IQC
  - `sourceBillType`: PUR_RECEIVE
  - `extContext.supplier`: 供应商编码 (用于匹配特定供应商的转严/放宽计划或免检控制)。
- **业务意义**: 确保物料入库成为可用库存前，必须经过 IQC 的检验批准，实现物理拦截。

### 4.2 MES → IPQC (制造过程检验)
- **触发节点 1 (首件检验)**: MES 中生产工单/制造指令单下达产线，操作工进行“开工/首件报工”时触发。
- **触发节点 2 (巡检/制程检验)**: MES 根据生产节拍（如每隔 2 小时）或特定工序的过站记录触发。
- **参数传递**:
  - `inspectType`: IPQC
  - `sourceBillType`: PROD_ORDER / PROD_OPERATION
  - `extContext.line`: 产线编码 / 机台编码。
  - `extContext.shift`: 班次代码。
- **业务意义**: 保证生产过程的稳定性，拦截批量产生废品的风险。

### 4.3 WMS/MES → FQC (成品/离线检验)
- **触发节点**: 生产车间将成品打包，推入成品区，WMS 生成“生产入库单” 或 MES 产品条码下线扫描时触发。
- **参数传递**:
  - `inspectType`: FQC
  - `sourceBillType`: PROD_INBOUND
  - `extContext.product_model`: 产品型号。
- **业务意义**: 产品在入库成为可销售成品前的终极把关。

### 4.4 ERP → OQC (出货检验)
- **触发节点**: 销售业务员在 金蝶云星空 (ERP) 中审核通过“销售发货单” / “出库申请单”，准备发货给客户前触发。
- **参数传递**:
  - `inspectType`: OQC
  - `sourceBillType`: SALES_DELIVER
  - `extContext.customer`: 客户编码。
- **业务意义**: 在物流发车前，基于客户特殊要求(CSR) 对发货批次进行出厂前的专项检验复核。

## 5. QMS 内部匹配及生成逻辑 (详细匹配引擎引擎)

当接收到外部接口的触发请求后，QMS 执行以下严密的管道式匹配漏斗逻辑：

### 第一步：组织与基础类型初筛
检索表中状态为 `已发布 (PUBLISHED)` 的 `qms_insp_plan` 数据。
进行基础条件硬匹配（SQL/内存过滤）：
- `org_id == Payload.orgId`
- `insp_type == Payload.inspectType`
- 找不到则输出：`IGNORED` (当前工厂该检验类型无可用的检验计划)。

### 第二步：物料与多维度精确匹配
对初筛结果进一步精滤：
- 必须满足 `material_code == Payload.materialCode` (或物料组规则)。
- 解析 `extContext`，与计划模型中的 `ext_context` 规则进行交叉比对：
  - 如果是 IQC 且 Payload 中存在 `supplier`，优先匹配包含该 `supplier` 的专项计划。若无，退网匹配 "空供应商 (代表全部供应商)" 的通用计划。
  - OQC 同理匹配 `customer`。

### 第三步：权重/优先级打分与唯一计划选定
经过前两步，若得到多条符合条件的计划。则按以下算法排序与选定：
1. `priority` 优先级数字越大/越小 (依系统设计)，优先级越高。
2. 约束条件越具体的权重越高（如 配置了特定供应商 S01 的计划，其权重高于适用于全部供应商的计划）。
3. 锁定系统中得分为第一名的 `Best_Insp_Plan`。

### 第四步：触发条件与动态策略评估 (Trigger Condition & Strategy)
锁定计划后，进行门槛和策略拦截：
- **单据类型拦截**: 校验 `Best_Insp_Plan.trigger_bill_type` 是否等于 `Payload.sourceBillType`。若不等于，抛出 `FAILED` (单据场景不符)。
- **动态免检/抽检策略拦截**: 
  - 根据系统里的历史检验记录和计划中配置的抽检规则计算。例如，如果计划设置“每5批抽检1批”或“前3批必须全检”。
  - 利用 `materialCode` + `extContext.supplier` 查询历史记录总批次数。
  - 判断出当前批次是否中签。如果不中签（如属于免检批），抛出 `IGNORED` (命中动态免检策略，无需检验)。

### 第五步：生成与任务实例化 (Instantiation)
通过所有拦截关卡后：
1. **获取最新检验标准 (Scheme Snapshot)**: 获取 `Best_Insp_Plan` 绑定的 `Scheme`，锁定所有检验条目、规格上限、下限、缺陷判定标准（深拷贝快照）。
2. **应用参数覆盖 (Parameter Overrides)**: 读取 `Best_Insp_Plan.param_overrides`（如有），针对该产品或供应商，利用微调阈值刷新快照中的公差参数。
3. **分派与创建 (Assignment)**: 读取计划配置的默认检验员、检验班组，生成唯一的 `TSK-IQC-XXXXX` 任务流水号，并写入 `qms_insp_task` 实体表。
4. **输出结果**: 返回给外部系统 `SUCCESS` 和对应的任务单号。同时写入触发日志记录。

## 6. 集成控制台前端页面实现功能

为了应对日常的调试、问题排查以及外部系统对接联调，专门在“**执行与控制 -> 集成控制台**”菜单下开发了相应的页面模块。

### 6.1 触发模拟器 (`TaskTriggerSimulator.vue`)
提供一个直接给 QMS "打一针" 虚拟触发请求的工具界面。
- **双模输入**: 支持可视化“表单模式”（供业务人员使用）和“JSON模式”（供研发调试对接报文）。
- **预设报文组**: 内置了一键填充 IQC, IPQC, FQC, OQC 典型的对接报文载荷，极大提升测试效率。
- **可视化链路渲染**: 核心功能是对匹配引擎后台处理逻辑的前端全链路展示。采用 `Timeline` 组件直观显示匹配漏斗中的每一步。即使最终没有生成任务，也能明确诊断出是由于“单据类型不匹配”、“找不到供应商特定的计划”还是“命中了免检策略”。无需查看后台日志，业务工程师亦可排查问题。

### 6.2 任务触发日志 (`TaskTriggerLog.vue`)
记录生产环境中发生的所有物理触发事件，起到追踪对接质量的作用。
- **丰富的检索**: 支持按照请求时间段、单据类型、检验类型、工厂ID、物料/批次号搜索，以及状态过滤（`SUCCESS` / `IGNORED` / `FAILED`）。
- **四项统计看板表盘**: 页面顶部提供“总请求数”、“成功生成”、“已忽略”、“匹配失败”的总揽汇总信息，系统对接健康度一目了然。
- **原因穿透展示**: 表格提供截断文本呈现初步判明原因，同时右侧“详情Drawer”展开展示 JSON 报文及带有场景图示的完整原因描述。
- **一键重放 (Replay Payload)**: 对于失败的或因策略被跳过的请求日志，提供一键“重放”功能。点击后系统自动将该条历史报文携带至“触发模拟器”中，并填充好 JSON 面板，以便研发或工程师立刻干预调试。
